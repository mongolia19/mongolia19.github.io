# 又炒概念？循环工程来了！

最近 AI 编程圈又冒出一个新词，叫“循环工程”，英文是 loop engineering。

照例，圈里先兴奋，随后一堆人开始转述，最后把一个本来不复杂的事情说得玄乎起来。干过几年工程的人其实不用被这个词吓到。它没那么新，也没那么神。你真把它拆开看，核心就一句话：别再手把手盯着 agent 一轮一轮输提示词了，改成把任务回路设计好，让 agent 自己在回路里跑。

这事为什么现在突然热起来？不是因为大家突然爱起名字了，而是因为模型能力刚好跨过了一条线。它现在不只是补代码、续几行函数了。它已经能读仓库、跑命令、改文件、执行测试、看报错、继续修。到了这一步，人类再坐在终端前一句一句指挥，反而开始变成系统里最慢的那个环节。

所以“循环工程”真正讨论的，不是 prompt 写得漂不漂亮，而是工程师怎么把一套原本靠人盯着的迭代过程，改造成一套可以持续运行、持续验证、持续收敛的系统。

## 这词听着新，骨子里其实很老

如果你做过 CI、做过批处理、写过值班 runbook、搭过异步任务框架，甚至只是认真维护过一条上线流水线，那你对这个事并不陌生。

以前我们写的是：

- 定时任务什么时候触发
- 出错后怎么重试
- 成功后怎么验收
- 失败了怎么报警
- 状态怎么持久化
- 半成品怎么清理

现在只不过把执行者从脚本、服务、人工操作，换成了 agent。

所以我更愿意把 loop engineering 理解成一句土话：你不是在“用 AI 写代码”，你是在“给 AI 修一条能自己跑的活路”。

这个视角一换，很多事情就明白了。

## 真正的难点，不在模型，在回路

很多人第一次接触 agent，脑子里想的是“它会不会自己想”“它能不能自己规划”“它是不是已经接近初级工程师”。这些问题不能说不重要，但都不在第一层。

第一层的问题其实是工程问题。

### 1. 触发器是谁

什么信号会启动一次 loop？

是新 issue 进来就触发。还是 PR 更新后自动复验。还是夜里定时扫 flaky tests。还是线上报错达到阈值后自动开工单。

没有触发器，loop 只是一个可演示的玩具。你手动点一下，它跑一遍，这不叫工程化。

### 2. 上下文从哪来

agent 看起来很聪明，但它最大的短板一直没变：它不天然知道你们团队的暗规则。

哪个目录不能碰，哪个脚本只是历史遗留，哪份 README 已经过期，为什么某段代码写得很丑但现在不能动，这些东西如果只存在于“老同事都知道”的层面，那 agent 必然踩雷。

所以 loop 真要跑起来，项目文档、仓库约束、任务说明、环境说明都得从“人类默契”变成“系统事实”。说白了，你得让 repo 对 agent 更友好，而不是只对新同事友好。

### 3. 验证器是不是够硬

这是整个 loop 最关键的东西。

agent 能改代码不稀奇。它改完以后，你凭什么相信它真的改对了，这才稀奇。

老工程师对这个坑都不陌生。很多问题并不是“没人会修”，而是“修完以后没人敢说已经好了”。一旦进入 agent 时代，这个问题会被放大，因为 agent 会比人更快地产生修改，也会更快地产生错改。

所以 loop 里必须有硬验证：

- 单元测试
- 集成测试
- 回归脚本
- 基准用例
- 格式检查
- 输出结构校验

没有这些，loop 不是闭环，只是连续生成。

### 4. 记忆是不是出上下文了

长流程任务最怕两件事：一件事是忘，另一件事是重复忘。

agent 今天试过什么、为什么失败、哪条路已经走不通、哪个中间结果可信，这些如果不写到外部存储里，下一轮大概率还会再踩一遍。

很多团队现在在 agent 身上重复了老毛病：以为上下文窗口就是记忆。不是。上下文是临时缓存，记忆得落盘。

所以 loop 一旦跨多轮、多小时，甚至跨天，状态持久化就是刚需，不是优化项。

### 5. 预算和刹车是不是先写好了

这是最容易被兴奋情绪淹没的一点。

很多人一想到 loop，就会本能地往“让它一直跑”去想。这个方向最危险。

真正能上线的 loop，第一件事不是加油门，是装刹车：

- 连续失败多少次就停
- 成本到多少就停
- 遇到哪些错误必须转人工
- 哪些目录只读
- 哪些命令禁用
- 哪些结果必须人工确认才能继续

工程系统从来不是靠信心跑的，是靠边界跑的。

## 为什么说这事会重新定义工程师分工

以前“写代码”是工程师最直接的产出。

现在这个定义开始松动了。

你会发现，真正值钱的工作慢慢往前挪了：

- 把目标定义清楚
- 把上下文组织清楚
- 把验证方式设计清楚
- 把失败分流规则写清楚
- 把 loop 跑出来以后持续调优

也就是说，工程师的主战场会越来越像控制台，而不是输入框。

这不代表人不写代码了。代码当然还要写。但很多重复性高、反馈明确、验证便宜的活，会越来越适合丢给 loop 去干。

比如下面这些：

- issue 初筛和分类
- 文档与代码漂移检查
- PR 的固定回归
- flaky test 的复现与收敛
- 重复性重构建议
- 失败构建的初步定位

这些活本来就不性感，但都很耗人，而且都很吃纪律。loop 先吃掉的，大概率就是这一类。

## 这波最容易死在哪

我看下来，接下来大多数团队如果想试 loop，死法不会太多，主要就那么几种。

### 第一种：拿 demo 当系统

看起来跑通一次，就以为工程化完成了。

但 demo 的世界里没有脏输入、没有并发、没有中断恢复、没有成本约束、没有历史包袱。真实仓库跟 demo 不是一个物种。

### 第二种：没有验收，只看感觉

“看起来像对了”“它自己说修好了”“日志里没再报那个错了”，这些都不能叫验收。

你要是让人类工程师这么提测，团队都不会放行。换成 agent 更不能放行。

### 第三种：上来就贪全自动

一上来就想让它自己接需求、自己拆任务、自己改代码、自己提 PR、自己发周报。这个阶段通常结束得很快，结束方式一般是成本爆炸或者结果没人敢用。

稳一点的路径反而很土：先找那些目标清楚、验证便宜、出错成本低的任务试。

### 第四种：把问题归咎于模型

很多时候不是模型不行，而是回路太烂。

上下文脏、约束没写、验证不全、状态没存、权限太大、失败不收口。这样的系统换一个更强的模型，也只是把错误跑得更快。

## 这个概念到底值不值得认真看

值得。

不是因为这个词本身有多高级，而是因为它确实把一件已经发生的事说透了：我们正在从“人直接驱动工具”，转向“人设计回路，回路驱动 agent，agent 驱动工具”。

这不是文字游戏，是工程形态在变。

以前一个强工程师的优势，更多体现在他写得快、修得准、排障稳。以后这些优势还重要，但会再往上加一层：谁更会搭回路，谁更会定边界，谁更会做验证，谁就更能把 agent 真正变成产能。

说得直接一点，loop engineering 不会淘汰工程师，但会区分两类工程师。

一类人还停留在“怎么把 prompt 写得更像咒语”。

另一类人已经开始想：

- 这条回路怎么触发
- 中间状态放哪
- 哪个结果算过关
- 哪种失败该停
- 哪段流程可以自动
- 哪段流程必须人兜底

后面这类人，才是真正在用 agent 做工程。

## 最后一句

所以，loop engineering 如果非要翻译成大白话，我给它一个不那么时髦、但更接地气的版本：

不是“让 AI 替你写代码”。

而是“把一套本来需要你反复盯着的活，修成一条它能自己跑、自己验、出错还能自己停下来的流水线”。

这个事一旦做成，价值不在于省了几次敲键盘。

价值在于，你终于把工程能力从个人手感，往系统能力推进了一层。
